이 문서는 스킬이 아니라 **레퍼런스 카탈로그**다. 검증·리뷰·리서치 작업을 설계할 때
아래 7개 패턴 중 작업 성격에 맞는 것을 골라 적용한다.

## 1. 적대적 검증 (Adversarial verify)
그럴듯하지만 틀린 결과를 반증으로 걸러낸다. 검증자에게 "동의하지 말고 틀린 곳을 찾아라"라고 지시.
- 적합: 버그 리포트, 사실 주장, 수치 검증, 보안 취약점 — 틀린 결과가 살아남으면 위험한 작업
- 적용처: [19-cross-verify.md](19-cross-verify.md), [08-pr-review.md](08-pr-review.md)

## 2. 관점 다양 검증 (Perspective-diverse verify)
같은 결과를 여러 번 보지 않고, 서로 다른 관점의 검증자가 서로 다른 실패 모드를 찾게 한다.
- 관점 예: 계산 정합성 / 출처 신뢰도 / 단위·기준 일관성 / 재현 가능성 / 보안 리스크 / 운영 리스크
- 적용처: 09-pr-check, 19-cross-verify

## 3. 심사 패널 (Judge panel)
여러 접근법을 독립적으로 만든 뒤 심사하고, 가장 강한 안에 다른 안의 장점을 합성한다.
- 적합: 아키텍처 선택, 전략 수립, 설계안 비교
- 적용처: [20-judge-panel.md](20-judge-panel.md)

## 4. 고갈까지 반복 (Loop-until-dry)
목표 개수를 채우는 게 아니라, 더 이상 새로운 발견이 안 나올 때까지 반복한다.
- 적합: 버그 감사, 리스크 탐색, 엣지 케이스 발굴
- 적용처: 보안 점검(09), 빌드 복구(lessons의 "양파 까기" — 한 겹 고치면 다음 겹)

## 5. 다중 모달 탐색 (Multi-modal sweep)
같은 대상을 문서별·키워드별·엔티티별·시간별·출처별로 나눠 탐색한다.
- 적합: 대규모 문서 분석, 코드베이스 감사 (bf/workspace, pact-conference 같은 멀티모듈)
- 적용처: 03-agents-md-generator, 11-context-engineering

## 6. 완전성 비평가 (Completeness critic)
최종 결과 직전에 "빠진 항목이 없는가"를 점검하는 역할을 둔다.
- 적합: 리서치, 감사, 보고서 — 망라성이 중요한 작업
- 적용처: 모든 산출형 프롬프트의 마지막 단계

## 7. 무성 절단 금지 (No silent truncation)
처리 범위를 줄였거나 일부만 검토했다면, 무엇을 생략했는지 반드시 밝힌다.
- AI가 일부만 보고 전체를 본 것처럼 말하면 사람이 잘못 판단한다.
- 이 lab의 정렬된 교훈: "에이전트 exit code를 맹신하지 말 것"(lessons) = 같은 정신
- 적용처: **전 프롬프트 공통 출력 말미에 "검토하지 못한 부분" 섹션 필수**

## 작업 유형별 권장 조합
| 작업 | 권장 패턴 |
|------|-----------|
| PR 리뷰 / 코드 리뷰 | 1 적대적 + 2 관점다양 + 6 완전성 |
| 보안·버그 감사 | 1 적대적 + 4 고갈까지 + 7 무성절단금지 |
| 기술 의사결정·설계 | 3 심사패널 + 1 적대적(1등 공격) |
| 교차 검증(QA) | 1 적대적 + 2 관점다양 (다른 모델로) |
| 대규모 코드베이스 분석 | 5 다중모달 + 6 완전성 |
