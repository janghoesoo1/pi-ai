# AI-네이티브 스타트업을 만드는 방법

> 진짜 변화는 시간 사용이 아니라 회사가 학습하고 반복하고 진화하는 **속도**에 있다.

---

## 핵심 요약

AI-네이티브 회사는 **운영 모델 자체**가 바뀐다.

- 적은 인원이 덜 조율하고
- 에이전트가 반복 작업을 실행하며
- 사람은 **방향, 취향, 관계, 검증, 책임**에 집중한다

모두가 같은 모델을 쓰는 환경에서 진짜 해자(moat)는 **규율(discipline)** -- 매주 업무를 매핑하고, 컨텍스트를 쌓고, eval을 쓰고, 루프를 돌리는 일관성이다.

### Open-loop vs Closed-loop 회사

> "오늘날 대부분의 Pre-AI 회사는 기본적으로 Open-loop으로 운영된다. [...] 이제 우리는 에이전트와 구현 방식을 회사의 의사결정에 통합해 Closed-loop 시스템으로 전환할 수 있다." -- Diana Hu, Stanford CS153

| 구분 | Open-loop (기존) | Closed-loop (AI-네이티브) |
|------|-----------------|------------------------|
| 정보 | 머릿속, DM, 복도 대화에 분산 | 모든 행동이 기계 판독 가능한 아티팩트 생성 |
| 피드백 | 결정 후 결과를 체계적으로 측정하지 않음 | 에이전트가 모든 아티팩트에 읽기 접근 |
| 오류 | 조용히 누적 | 에이전트가 다음 행동을 제안하고 자가 치유 |
| 결과 | 정보 손실(lossy) | 직원 1인당 $1~2M 매출 (전통 대비 10배+) |

> 로보틱스의 PID 컨트롤러를 조직 설계에 적용한 것이다.

## 구조적 전제: 조직 재설계를 강제하는 4가지 힘

에이전트가 조직을 바꾸는 이유는 단순히 "효율적"이어서가 아니라 네 가지 물리적 속성이 기존 전제를 무너뜨리기 때문이다.

| 속성 | 사람 | 에이전트 | 함의 |
|------|------|---------|------|
| **비용** | 급여+복리+오피스+관리+채용 | 토큰 단가 | 정보 처리 작업의 비용 격차 10~100배 |
| **속도** | 시간~일 | 밀리초 | 가설 검증 주기가 일주일에서 분 단위로 압축 |
| **대역폭** | Dunbar's number (~150 관계) | 무제한 | 한 사람이 수천 에이전트를 동시 지휘 가능 |
| **가용성** | 영업시간, 시차, 휴식 | 24/7 | 연속 운영이 예외가 아닌 기본값 |

> 시간이 지날수록 이 격차는 복리로 누적되어 경쟁 우위가 된다.

### 핵심 전환: 인원수 최대화에서 토큰 사용 최대화로

> "핵심 전환은 인원수 최대화에서 토큰 사용 최대화로의 전환이다. 최고의 회사는 가장 큰 급여 목록이 아니라 AI 토큰 예산을 가장 효율적으로 사용하는 회사다." -- Diana Hu

- 월 $50,000 AI API 비용이 5명 이상 엔지니어($150K+ 연봉 + 복리 + 관리 오버헤드)를 대체
- 초기 창업자는 레거시 없이 처음부터 AI 중심으로 설계할 수 있어 압도적 유리

## 두 스타트업의 대비

같은 시장, 같은 달에 창업한 두 회사의 오전 9시.

### 회사 A (기존 방식)
- 운영 리드가 밀린 지원 티켓 처리
- 분석가가 지난주 대시보드를 다시 만드는 중
- 창업자는 아무도 해결 못 한 고객 통화 건으로 스탠드업 중
- **모두 어제의 문제를 수습하느라 제품은 정체**

### 회사 B (AI-네이티브)
- 그 모든 일을 **밤새 에이전트가 처리** 완료
- 티켓 분류, 대시보드 갱신, 이탈 위험 표면화 완료
- 창업자는 이미 문제를 고치고 팀과 제품을 개선 중

핵심 차이는 **학습 속도**. 회사 B가 매일 더 빨리 배워 수주, 수개월, 수년에 걸쳐 레버리지가 복리로 누적된다.

## 전환 6단계

```
1. 업무 매핑 (Map The Work)
2. 컨텍스트 시스템 구축 (Build The Context System)
3. 가장 단순한 자동화 선택 (Choose The Simplest Automation)
4. 스킬과 eval 인코딩 (Encode Skills And Evals)
5. 팀을 AI-네이티브로 (Make The Team AI-Native)
6. 스타트업을 피드백 루프로 운영 (Run As A Feedback Loop)
```

---

## 1단계: 업무 매핑 (Map The Work)

지난 2주간 반복된 업무를 모두 나열한다.

**나열 대상 예시:**
고객 통화 노트, 리드 리서치, 아웃바운드 초안, 지원 분류, 제품 QA, 온보딩, 릴리스 노트, 투자자 업데이트, 주간 지표, 버그 재현, 채용 스크리닝, 인보이스 검토, 경쟁사 모니터링

> 반복되면 인코딩 후보. 대부분 창업자 캘린더에 20~40개 항목이 존재한다.

### 자율성 레벨 분류

| 레벨 | 설명 | 예시 |
|------|------|------|
| **L1** | 사람 전용 | 전략적 결정, 최종 채용, 큰 환불, 법적 서명, 이사회 커뮤니케이션 |
| **L2** | AI 준비 + 사람 승인 | 투자자 업데이트 초안, 계약 레드라인, 가격 페이지, 지원 매크로 |
| **L3** | AI 실행 + 사람 감독 | 인바운드 분류, 미팅 노트 라우팅, 리드 보강, 테스트 생성 |
| **L4** | 명확한 한계 안에서 자율 | 경쟁사 모니터링, 야간 리포트, 인보이스 추출, 이상 탐지 |

### 지루한 워크플로가 보통 승리한다

주간 전략 메모보다 **매일 반복되는 지원 태깅**이 10배 자주 실행되어 더 많은 시간을 회수하고 깨끗한 정답 데이터를 제공한다.

**첫 공략 대상 선택 기준:**
- 빈번하고
- 측정 가능하고
- 되돌릴 수 있고
- 실제 병목과 연결된 작업

### 자동화하면 안 되는 업무

- 드물거나, 불명확하거나, 높은 신뢰가 필요하거나, 불안정한 업무
- 좋은 결과물이 무엇인지 설명 못 하면 인코딩 준비가 안 된 것
- 한 번의 잘못된 출력이 고객 관계를 해칠 수 있으면 천천히 진행

> C.H. Robinson 팀은 하루 1만 건 이메일 분류를 L4로 밀었다가 감독이 불가능해 L2로 복귀. 물량이 오분류 비용을 가렸다.

### 시작 구성: 1페이지 + 3개 워크플로

| 영역 | 워크플로 예시 |
|------|-------------|
| 개인 | 인박스 분류, 데일리 브리프, 투자자 업데이트 초안 |
| 고객 대면 | 통화 종합, 티켓 분류, 리드 보강 |
| 내부 | 테스트 생성, 인보이스 추출, 주간 지표 서술 |

> 동시에 너무 많은 실험은 주의를 분산시켜 20개의 미완성 파일럿을 만드는 가장 흔한 실패 모드를 유발한다.

## 2단계: 컨텍스트 시스템 구축 (Build The Context System)

컨텍스트는 AI-네이티브 스타트업의 **운영 기억(operating memory)**.

> 모델은 교체 가능하지만 컨텍스트는 진짜 회사만의 자산.
> 공동창업자처럼 일하는 에이전트와 혼란스러운 임시직처럼 일하는 에이전트를 가른다.

출력 재작성에 검토보다 더 많은 시간이 든다면, 문제는 프롬프트나 모델이 아니라 **에이전트가 회사를 충분히 모르는 것**이다.

### 가독성(Legibility): 회사를 AI에게 질의 가능하게 만들기

> "조직 전체가 AI에 가독적(legible)이어야 한다. 모든 중요한 행동은 회사 중심의 지능이 학습하고 자기 개선에 사용할 수 있는 아티팩트를 생산해야 한다."

가독성의 3요소:

**1) 아티팩트 캡처**
- 모든 중요한 행동이 기계 판독 가능한 아티팩트를 생산해야 한다
- 모든 회의를 AI 노트테이커로 녹취/전사
- DM과 산발적 이메일을 줄이고 추적 가능한 채널 선호
- "운영적으로 중요한 사실은 누군가의 기억이나 회의 후 복도 대화에만 존재하면 안 된다"

> "Slack에서 대화하는 것이 엘리베이터에서 대화하는 것보다 낫다. 컴퓨터가 Slack은 볼 수 있기 때문이다." -- Turing Post

**2) 데이터 중앙화**
- 매출, 영업 파이프라인, 엔지니어링 진행, 채용 현황, 운영 지표를 종합 대시보드로
- 고객 피드백(이메일, 지원 티켓, 도구)을 중앙 시스템에 자동 투입
- GitHub 커밋/PR 로깅, 영업 통화 녹취, 일일 스탠드업 캡처, 전략 문서 접근

**3) 시스템 상호 연결**
- 모든 데이터 소스를 연결해 조직 AI가 완전한 그림을 그릴 수 있게
- Linear 티켓 + Slack 엔지니어링 채널 + 고객 피드백 + 전략 계획 + 영업 통화 + 스탠드업 녹취에 접근하는 에이전트는 실제 배포 내용과 고객 니즈 충족도를 분석 가능
- "AI 시스템에 인간 직원에게 주는 만큼의 컨텍스트를 제공하라"

### 주간 진단법

대표 작업 하나를 워크스페이스 컨텍스트만 가진 새 에이전트에게 주고 다음 행동 3가지를 요청:
- 강한 제안 2개 이상 -> 컨텍스트가 제 역할
- 일반적 답변 3개 -> 컨텍스트가 빈약 (프롬프트로 구제 불가)

### Git 저장소 기반

모든 팀원과 에이전트가 읽을 수 있는 **공유 Git 저장소 하나**로 시작.

**7일차 워크스페이스 구조:**

```
/
  CLAUDE.md
  context/
    company.md
    product.md
    customers.md
    lessons.md
  GTD.md
```

> 손으로 쓴 40~60줄로 유지. 피해야 할 것의 촘촘한 목록이 생성된 400줄의 잡문보다 우수하다.

### 권한 경계로 분할

성장 시: 공유 회사 저장소 + 기능별 비공개 저장소 (영업, 제품, 엔지니어링, 재무, 지원)

> Block의 내부 하네스 Goose는 역할별 스코프로 동일 아티팩트 스트림을 읽는다.
> 스코프가 어긋나면 공개 발언과 비공개 거래 컨텍스트를 섞기 시작. 경계가 매우 중요하다.

### 시스템에 들어가는 3종 데이터

#### 1) 커넥터 -- 외부 데이터 수집

SaaS, API, MCP 서버, 이메일, 캘린더, CRM, 지원, 분석, GitHub, Linear, Stripe 등

- 모든 커넥터에 **식별자, 스코프 권한, 감사 로그, 관리되는 자격증명** 필요
- 없으면 최대 보안 구멍

> Anthropic의 MCP 코드 실행 작업은 서버 폴더 파일시스템 패턴으로 컨텍스트 사용을 약 15만 토큰에서 약 2천 토큰으로 **98.7% 절감**.

#### 2) 회사가 생성하는 데이터

스펙, 고객 요약, 결정, 교훈, 프로젝트 노트, 운영 규칙. 기본은 Markdown.

**가장 중요한 규칙: 원본(raw)과 정제본(distilled) 분리**

| 구분 | 예시 |
|------|------|
| 원본 | 통화 녹취 전문 |
| 정제본 | 결정, 고객 이의, 후속 담당자, 갱신 위험 |

에이전트가 실제 조회하는 대상은 정제본이다.

> 결정 로그는 한 줄에 하나씩 append-only로 유지해 추론이 결론과 함께 따라가게 한다.

#### 3) 데이터베이스와 스트림

- Markdown으로 맹목 복사하지 말고 DB를 원천으로 유지
- 에이전트에 **제한된 읽기 사용자** 부여
- 스키마를 에이전트용 파일(`data-models/postgres.md`)에 문서화
- 기본: 에이전트가 프로덕션 데이터를 삭제 불가하게 설정

> 2025년 중반 Replit 사건 -- 코딩 에이전트가 세션 중 프로덕션 DB를 삭제. 프롬프트 지시는 보안 경계가 아니다.

### 고급: 출처 추적

- 모든 요약은 출처(녹취, 티켓, 커밋, 인보이스, DB 행)로 되짚을 수 있어야 한다
- 출처가 없으면 검증 불가한 그럴듯한 요약이 쌓이고 첫 오답에서 에이전트 전체에 대한 **신뢰가 붕괴**
- 출처가 있으면 에이전트는 감사 가능해져 팀원이 클릭 한 번으로 소스를 확인

### AI-네이티브 도구 선택 원칙

> "가시성(visibility)과 이식성(portability)으로 도구를 선택하라" -- Turing Post

- 에이전트에 데이터를 노출하는 도구를 선호
- 컨텍스트에 기록, 소유권, 접근 제어, 명확한 상태(초안/승인), 현행/폐기 구분이 있어야 한다
- 에이전트 시스템은 단순하게 유지 -- 복잡성은 도움이 된다는 증거가 있을 때만 추가

## 3단계: 가장 단순한 자동화 선택 (Choose The Simplest Automation)

> 모든 워크플로를 에이전트로 만들지 말 것.
> 최고의 시스템은 스크립트, AI 보조 사람, 결정론적 워크플로, 에이전트의 **혼합**이다.

### 도구별 적용

| 도구 | 적합한 경우 | 예시 |
|------|------------|------|
| **스크립트** | 결정론적 단계 | 리포트 내보내기, CSV 변환, 테스트 실행, JSON 검증 |
| **AI 보조 사람** | 판단이 필요한 출력 | 투자자 업데이트, 가격 카피, 계약 노트 |
| **워크플로** | 단계가 사전에 알려진 경우 | 통화 수집 -> 요약 -> 이의 추출 -> CRM 노트 -> 후속 생성 |
| **에이전트** | 경로를 미리 정할 수 없는 경우 | 프로덕션 버그 조사, 시장 리서치, 비정상 지원 케이스 |

### 하네스(Harness) -- 6단계 안전 레이어

```
1. Preflight  -- 토큰 쓰기 전 컨텍스트와 권한 확인
2. Plan       -- 작업 분해 및 제안 단계 노출
3. Approve    -- 사람 또는 판정 모델 게이트가 나쁜 계획을 실행 전 차단
4. Execute    -- 작업 실행
5. Verify     -- 테스트, 스키마, 루브릭, 예시로 출력 검증
6. Log        -- 다음 반복이 정답을 갖도록 실행 기록
```

### 가드레일은 코드와 설정에 (프롬프트가 아님)

> "프로덕션 데이터를 삭제하지 마"라는 프롬프트는 보안 경계가 **아니다**.

**협상 불가 항목:**

- 실행/일일 비용 상한
- 재시도 상한
- 최대 도구 호출 깊이
- 에이전트별 스코프 자격증명
- 승인 없는 프로덕션 쓰기 금지
- 코드 자동 머지 금지
- 전 함대 킬 스위치

## 4단계: 스킬과 eval 인코딩 (Encode Skills And Evals)

> 반복 업무를 스킬로 인코딩하고 eval로 채점하는 것이
> 회사를 매주 조금씩 **복리로 성장**시키는 엔진이다.

### 스킬 = 반복 작업을 위한 재사용 지침 + 예시

**모든 스킬에 필요한 형태:**

| 항목 | 설명 |
|------|------|
| Scope | 언제 이 스킬이 적용되는가 |
| Inputs | 무엇을 받는가 |
| Load | 어떤 컨텍스트를 로드하는가 |
| Steps | 절차 |
| Output | 출력 형식 |
| Examples | 작동 예시 |
| Escalate | 언제 사람에게 넘기는가 |
| Owner | 누가 유지하는가 |
| Eval | 어떻게 채점하는가 |

> 무엇을 받고, 반환하고, 언제 도움을 청하고, 누가 유지하는지 적혀 있지 않으면 그것은 스킬이 아니라 긴 프롬프트다.

### 에이전트 프리미티브와 회사 구조의 대응

에이전트 시스템의 구성 요소는 회사 조직의 역할과 1:1로 대응한다:

| 에이전트 프리미티브 | 회사 대응물 |
|-------------------|-----------|
| **Skills** | 직원 (각 스킬 = 한 직원의 역량) |
| **Resolver** | 조직도 (올바른 스킬에 작업 라우팅) |
| **CheckResolvable** | 감사/컴플라이언스 (고아 작업 방지) |
| **Memory (3계층)** | 조직 지식 |
| **Evals** | 성과 평가 |

### Skillify: 스킬을 신뢰할 수 있게 만드는 10단계

> "스킬을 작성하고 코드를 쓰는 건 10단계 중 2단계에 불과하다. 나머지 전부는 이 지저분한 시스템이 -- 완벽한 코드보다 인간 시스템에 가까운 -- 여전히 작동하도록 만드는 것이다." -- Garry Tan

| 단계 | 무엇 | 상세 |
|------|------|------|
| 1 | **SKILL.md (계약)** | YAML frontmatter (name, version, triggers[], tools[], mutating 플래그) + Contract/Phases/Output Format |
| 2 | **결정론적 스크립트** | 실제 코드. LLM 호출 0건, <100ms 응답 |
| 3 | **단위 테스트** | fixture 데이터 기반 Vitest 테스트. Unicode 손실, 윤년 버그, 누락 배열 검출 |
| 4 | **통합 테스트** | 실제 데이터(라이브 캐시, 실제 JSON) 대상 실행. 타임존, 자정 걸침 이벤트 검증 |
| 5 | **LLM 평가** | 최소 3케이스 eval (happy / edge / failure). LLM 호출이 있으면 필수 |
| 6 | **Resolver 트리거** | RESOLVER.md에 트리거 추가. 사용자가 **실제로 입력하는** 패턴 사용 |
| 7 | **Resolver 평가** | 올바른 스킬이 발동되는지 테스트. 라우팅 혼란 방지 |
| 8 | **CheckResolvable + DRY 감사** | 주간 감사로 AGENTS.md -> SKILL.md -> script/cron 체인 검증. 도달 가능성, MECE 중복, 갭 탐지. Garry의 시스템에서 40개 스킬 중 6개(15%)가 "dark"(도달 불가)였음 |
| 9 | **E2E 스모크 테스트** | 전체 시스템 테스트 |
| 10 | **Brain Filing 규칙** | 스킬이 지식 베이스에 쓸 때의 목적 디렉토리 규칙. 명시적 규칙 적용 전 13개 중 10개 스킬이 잘못 분류됨, 적용 후 오분류 0건 |

**아키텍처 원칙: "Fat Skills, Fat Code, Thin Harness"**
- **Skills** = 풍부하고 상세한 재사용 워크플로 모듈 (FAT)
- **Code** = 상당한 로직을 가진 결정론적 스크립트 (FAT)
- **Harness** = 라우팅만 하는 최소한의 오케스트레이션 레이어 (THIN)

> "모델은 그저 엔진이고, 나머지 전부가 자동차다."

### 3계층 메모리 시스템

| 계층 | 역할 |
|------|------|
| **1. Knowledge Wiki** | 기본 정보 저장 |
| **2. Vector Search + RRF Fusion + Backlinks** | 검색 강화와 연결 |
| **3. Graph DB + Epistemology System** | 출처 추적이 있는 지식 그래프. "누군가 X를 믿었고 그것을 현실로 만들었다"는 과정 추적 |

### 창업자 친화 스킬

| 스킬 | 내용 |
|------|------|
| 고객 통화 종합 | 녹취에서 이의, 기능 요청, 위험, 약속, 다음 행동 추출 |
| 인박스 분류 | 투자자/고객/채용/운영 분류 + 앞 3종 답장 초안 |
| 투자자 업데이트 | 지표와 결정에서 서술 작성, 양쪽 인용 |
| 가격 페이지 분석 | 라이브 페이지를 고객 이의 로그와 비교 |
| 주간 지표 서술 | 무엇이 바뀌고, 깨지고, 점검할지 설명 |
| 테스트 생성 | 스펙을 테스트와 초안 PR로 전환 |

### "취향을 실행 가능하게 만든 것이 eval이다"

> "코드 배포 비용은 0에 수렴하지만, 무언가를 좋게 만드는 취향과 좋고 나쁨을 가르는 능력은 0에 수렴하지 않는다. Eval은 취향을 실행 가능하게 만든 것이다." -- Diana Hu

- **취향(taste)** = 위임할 수 없는 인간의 품질 판단 능력. 자동화 불가.
- **Eval** = 취향이 에이전트 속도로 실행되도록 **컴파일**하는 메커니즘.
- **범용 벤치마크는 함정**: MMLU는 제품이 사용자를 실제로 돕는지 알려주지 않는다. **제품 고유 eval**이 필요하다.
- "eval은 사람이 트레이스에 들어가서 옳고 그름을 라벨링하고 교훈을 skillify할 때만 존재한다."

### eval의 3개 레이어 (순서대로 누적)

```
Layer 1: 손 라벨링 정답
  사람이 실제 예시에서 좋은 출력이 무엇인지 표시

Layer 2: 결정론적 검사
  비용 제로에 명확한 판정
  - 스키마 유효성
  - 소스와 일치하는 숫자
  - 해석되는 링크
  - 존재하는 인용
  - 통과하는 테스트

Layer 3: LLM 판정
  결정론적 검사가 닿지 못하는 부분만
  - 글 품질, 감정, 브리프 부합 여부
  - 작고 빠른 모델로 충분
  - 신뢰 전 사람 표시 예시로 보정 필요
```

### 고급: Cross-modal Eval

같은 입력을 **여러 프론티어 모델(Opus, GPT-5.5, DeepSeek-V4)**에 통과시켜 교차 평가:
- 복수 모델의 합의가 단일 모델이 놓치는 블라인드 스팟을 드러냄
- 점수가 자동으로 원래 서브에이전트에 피드백되어 반복 개선
- 단일 모델 의존도를 낮추고 eval의 견고성을 높임

### 사례 연구: 고객 통화 종합

1. 과거 통화 30건을 revenue lead가 표시 (중요한 사실, 이의, 위험, 후속)
2. 결정론적 확인: 이름, 계약상 금액, 후속 날짜의 주(week) 정확성
3. LLM 판정: 브리프가 통화처럼 들리는지
4. 약 50회 실행 후 보통 같은 두 가지가 깨짐:
   - 3명 이상 통화에서 화자 추적 실패
   - 서로 다른 두 이의를 하나로 병합
5. 클러스터 단위로 고쳐 일관 동작까지 재작성

### eval 성숙 5단계

```
1. 예시 하나 수동 점검
2. 작성된 루브릭으로 소수 사례 채점
3. 그 루브릭을 과거 20~300건에 적용
4. 에이전트가 본 적 없는 홀드아웃 세트로 테스트
5. 스킬이 움직이려던 비즈니스 지표 추적
```

### 출시 후 매주 추적할 6개 숫자

| 지표 | 의미 |
|------|------|
| 수용률 (acceptance rate) | **핵심 지표**. ~70% 미만이면 자율성 레벨 올릴 준비 안 됨 |
| 오버라이드율 | 사람이 결과를 덮어쓴 비율 |
| 실행당 비용 | 에이전트 실행 1회 비용 |
| 사이클 타임 | 작업 완료까지 걸린 시간 |
| 검토 시간 | 사람이 검토에 쓴 시간 |
| 사건 수 | 에이전트 관련 사고 건수 |

> 수용률이 낮을 때 프롬프트 재작성은 거의 정답이 아니다.
> 보통 네 가지 중 하나:
> 1. 런타임에 컨텍스트 추가
> 2. 스킬 범위 좁히기
> 3. 파일에 작동 예시 추가
> 4. 맡지 말았어야 할 케이스에 대한 명확한 에스컬레이션 규칙 작성

## 5단계: 팀을 AI-네이티브로 (Make The Team AI-Native)

### 창업자가 먼저 시작

캘린더, 인박스, Slack에서 밤새 끌어온 결과를 팀에게 라이브로 보여준다:
- 아침 브리프
- 어제 통화의 고객 종합
- 스펙 대비 에이전트가 연 테스트 PR
- 지표 팩으로 만든 투자자 업데이트 초안

> Jack Dorsey는 2025년 내내 매일 아침 수 시간 이 도구들을 직접 다룬 뒤 Block을 재편.
> 결정은 에이전트를 직접 써본 리더십에서 나왔다.

### 3가지 새로운 역할

중간 관리자 레이어가 축소되고, 세 가지 역할로 수렴한다. Diana Hu(YC), Dorsey/Botha(Block/Sequoia)가 독립적으로 같은 결론에 도달:

| 역할 | 정의 | 핵심 |
|------|------|------|
| **IC (빌더)** | 모든 사람이 무언가를 직접 만든다 | 비기술 인력도 도구로 직접 빌드. 회의에 발표 자료가 아닌 작동하는 프로토타입만 들고 옴 |
| **DRI (직접 책임자)** | 한 사람, 하나의 명확한 결과, 완전한 책무 | 프로세스나 사람 관리가 아닌 교차 기능 결과를 소유. 90일 사이클로 운영 |
| **Player-Coach** | 빌딩과 사람 육성을 결합 | 여전히 코드 작성/모델 빌드/디자인. 상태 회의, 정렬 세션, 우선순위 협상 없음 |

> "중간 관리 레이어의 영구적 필요는 없다. 기존 계층이 하던 나머지 모든 것은 시스템이 조율한다." -- Block

### 온보딩은 산출물로 종료

첫 세션에서 **그날 배포 가능한 산출물**을 들고 나와야 한다:
- 정리된 고객 브리프, 지원 매크로, 테스트 PR, 가격 페이지 비평

> 실제 작업을 만들지 못하는 교육은 다음 주면 잊힌다.
>
> Ramp의 Glass 도구는 매 온보딩 세션이 새 사람의 스킬/아티팩트 1개를 공유 라이브러리에 추가하며 끝나는 규칙으로 3개월 만에 일일 사용자 20명에서 약 700명으로 증가.

### 사람과 에이전트의 역할 분담

| 사람 | 에이전트 |
|------|---------|
| 시스템 설계 | 실행 |
| 관계 소유 | 데이터 처리 |
| 출력 판단 (취향) | 반복 작업 |
| 책임 | 스케일 |
| 직관, 문화적 맥락, 신뢰 역학 | 연속 운영, 정보 라우팅 |

> 좁은 작업만 돌리는 팀원은 이 모델에서 노출된다.
> 판단을 지침, 예시, eval로 바꾸는 사람은 이전보다 더 가치 있다.

### 채용도 변화

채용 시 잡지식 대신, **주어진 시간 안에 손으로 못 끝낼 큰 프로젝트**를 주고 에이전트를 어떻게 지휘하는지 관찰한다.

| 역할 | 과제 | 핵심 |
|------|------|------|
| 분석가 | 3시간 안에 실제 리포트 (소스 수집 -> 증거 추출 -> 브리프) | 시간 내 손으로 완료 불가 |
| 엔지니어 | 하루 안에 제품 표면 복제 또는 내부 도구 (테스트 포함) | 에이전트 지휘 능력 |
| 그로스 | 50~100개 회사 대상 시장 맵 + 캠페인 계획 | 판단 + 취향 + 복구력 |

## 6단계: 피드백 루프로 운영 (Run As A Feedback Loop)

AI-네이티브 스타트업은 **매주 운영 체계를 개선**한다. 처음으로 돌아가 다시 시작.

### 주간 리뷰에서 다룰 것

- 에이전트가 한 일
- 사람이 오버라이드한 지점
- 실패한 eval
- 누락된 컨텍스트
- 좁혀야 할 스킬
- 죽일 워크플로
- 자율성 레벨을 올릴 워크플로

### 동시에 두 루프 실행

```
이너 루프 (기존 작업 개선)
  - 실행당 비용 감소
  - 사이클 타임 감소
  - 사건 감소
  - 아티팩트당 검토 시간 감소

아우터 루프 (다음을 탐색)
  - 신규 고객 세그먼트
  - 제품 아이디어
  - 가격 변경
  - 파트너십
  - 경쟁사 움직임
  - 규제 변화
  - 이탈 위험
  백그라운드 에이전트가 후보를 24시간 공급, 사람이 추격 대상 선택
```

### 소프트웨어 팩토리 (이너 루프의 핵심)

사람이 스펙과 테스트를 쓰고, 에이전트가 그에 맞춰 구현. 결정론적 검사는 스스로 돌고 **머지 전 사람이 검토**.

**TDD 기반 AI 소프트웨어 팩토리:**
- 사람이 스펙 + 성공 기준(테스트 케이스)을 작성
- AI 에이전트가 **모든 구현 코드**를 생성
- AI가 반복하며 테스트를 실행하고 모든 테스트 통과까지 정제
- 일부 회사는 **손으로 쓴 코드 없이 스펙과 테스트 하네스만** 있는 저장소를 운영

> StrongDM의 AI 팀은 인간 코드 작성과 리뷰를 완전히 제거했다.
> 이것이 "1000x 엔지니어"를 만든다 -- 뛰어난 개인이 아니라 AI 에이전트로 둘러싸인 엔지니어.

**시작 대상:**
- 테스트 생성
- 의존성 업그레이드
- 마이그레이션
- 내부 도구
- 통합 스캐폴드
- QA 스크립트
- 보안 자동 수정

**두 가지 하드 룰:**
1. 어떤 것도 **자동 머지되지 않음**
2. 어떤 에이전트도 **프로덕션에 쓰지 않음**

> Cursor조차 대규모 자율 클라우드 에이전트를 돌리면서도 2026년 초까지 머지에 사람 검토 게이트를 유지했다.
> 이 게이트가 나머지를 안전하게 확장하게 한다.

### 회사 차원 자기 개선의 핵심 = eval 작성 능력

1. 수백 개 예시를 좋음/나쁨으로 손 라벨링
2. eval을 만들어 프롬프트 진화 루프에 연결
3. GEPA, DSPy 같은 프레임워크가 프롬프트 변이를 제안
4. eval이 라벨 세트에서 순위를 매겨 승자를 배포, 반복

> 창업자는 eval을 쓰고 실패 클러스터를 읽되, 진화된 프롬프트는 쓰지도 읽지도 않는다.

> 발목을 잡는 것은 에이전트 역량이 아니라 **좋음의 기준을 인코딩할 수 있느냐**.
> 코딩은 도움이 되나 병목이 아니며, 출력을 신뢰성 있게 좋음/나쁨으로 표시할 수 있는 도메인 전문가가 전체 루프를 돌릴 수 있다.

## 심화: 회사 = Mini-AGI (Block의 4계층 아키텍처)

> "Block에서 우리는 기본 전제에 의문을 제기하고 있다: 조직이 계층적으로 구성되어야 하며 인간이 조율 메커니즘이어야 한다는 전제. 대신 우리는 계층이 하던 일을 대체하려 한다." -- Jack Dorsey & Roelof Botha, "From Hierarchy to Intelligence" (2026.03)

### 왜 계층인가

계층은 2,000년간 **정보 라우팅 프로토콜**이었다:
- 로마군 (8 -> 80 -> 480 -> 5,000 구조)
- 프로이센 참모본부 ("중간 관리자" 이전의 중간 관리)
- 미국 철도 (Daniel McCallum의 1850년대 조직도)
- Taylor의 과학적 관리법
- 맨해튼 프로젝트의 교차 기능 팀
- McKinsey 매트릭스 조직 (1959)
- Spotify 스쿼드, Zappos 홀라크라시, Valve 수평 구조 (모두 규모 확대 시 계층으로 회귀)

지배 제약: 리더가 효과적으로 관리할 수 있는 인원은 3~8명. 그 임계치를 넘으면 레이어 추가. 각 레이어는 지연, 왜곡, 정치를 추가한다.

> "2,000년의 조직 혁신은 이 트레이드오프를 깨뜨리지 않고 우회하려는 시도였다."

### 4계층 아키텍처

AI가 처음으로 전체 비즈니스의 지속적으로 업데이트되는 모델을 유지할 수 있게 되면서, 그 레이어들을 대체할 수 있다.

| 계층 | 무엇 |
|------|------|
| **1. Capabilities** | 원자적 금융 프리미티브: 결제, 대출, 카드 발행, 뱅킹, BNPL, 급여. UI 없음. 신뢰성/컴플라이언스/성능 스펙만 있는 빌딩 블록 |
| **2. World Model** | (a) **회사 모델** -- 모든 결정, 코드 커밋, 디자인 플랜, 워크플로 아티팩트를 실시간 운영 컨텍스트로 수집 (b) **고객 모델** -- 양방향 거래 데이터에서 고객/머천트별 이해 |
| **3. Intelligence** | 특정 고객에게 특정 순간에 맞는 솔루션을 Capabilities에서 프로액티브하게 조합. 예: 머천트의 현금 흐름이 시즌 저점 전에 타이트해지는 것을 감지 -> 단기 대출을 사전 조립 |
| **4. Interfaces** | Square, Cash App, Afterpay, TIDAL -- 전달 표면. 중요하지만 가치가 만들어지는 곳은 **아님** |

> "지능 레이어가 솔루션을 조합하려다 해당 Capability가 없어서 실패하면, 그 실패 신호 자체가 미래 로드맵이다."

전통적 제품 로드맵을 고객 현실이 주도하는 **시스템 생성 백로그**로 대체한다.

### 지능은 중심에, 사람은 엣지에

> "지능은 시스템에 살고, 사람은 엣지에 있다. 엣지는 지능이 현실과 접촉하는 곳이다." -- Dorsey

엣지의 사람이 하는 일:
- 모델이 아직 도달하지 못하는 곳에 닿는다
- 모델이 감지하지 못하는 것을 감지한다: 직관, 문화적 맥락, 신뢰 역학, "방 안의 분위기"
- 모델이 내리면 안 되는 판단을 내린다: 윤리적 결정, 신규 상황, 고위험 순간
- 지능을 고객 결과 방향으로 지속적으로 정렬한다

> "CEO로서 내 일은 [지능을] 올바른 결과라고 생각하는 방향으로 끊임없이 정렬하는 것이다."

### 코파일럿 vs 지능 중심

- 모든 직원에게 코파일럿 지급 = 기존 구조를 약간 개선. **점진적**.
- 회사를 지능으로 구축 = 구조 자체 변경. **구조적**.

> "회사를 보면 그것은 하나의 지능이다. 다만 가장 효율적이거나 정보 흐름에서 가장 손실이 적은 방식으로 구조화되어 있지 않았을 뿐이다."

## 심화: Digital FTE와 수익화

### Digital FTE 정의

> "AI-네이티브 스타트업 -- 처음부터 Digital FTE, 평가, 공유 메모리, 클로즈드 루프 피드백을 중심으로 설계된 회사. Digital FTE를 고객에게 판매하거나 내부적으로 운영한다." -- AI Agent Factory

- **Digital FTE** = 정규직 동등물로 포지셔닝된 AI 에이전트
- **인간 노동의 완전 적재 비용** (~$50,000+ 연봉)에 대비하여 가격 책정. $50 SaaS 구독이 아님
- 수익화 핵심: IT 예산(소프트웨어)은 작고 엄격하지만, HR/부서 예산(급여)은 10배 이상 크다

### 2계층 모델

| 계층 | 주체 | 역할 |
|------|------|------|
| **Edge Layer** | 인간 주체(principal) + 그들의 대리 에이전트(identic delegate) | 창의적, 문화적, 윤리적 결정. 취향. 책무. 감독 |
| **AI Workforce Layer** | Digital FTE | 실행. 조율. 정보 라우팅 |

### 제조 / 패키징 / 수익화 루프

1. **제조(Manufacture)**: Agent Factory 프로세스로 Digital FTE를 스펙화 (skill-resolver-memory-evaluation 루프)
2. **패키징(Package)**: 배포를 위한 래핑 (컴플라이언스, 가드레일, 통합)
3. **수익화(Monetize)**: 소프트웨어가 아닌 인건비 대비 가격 책정

> "팩토리는 시장에 나가기 전에 완료하는 단계가 아니다. 팩토리가 곧 판매하는 것이자 되고 있는 것이다."

### 4가지 수익화 모델

| 모델 | 방식 | 적합 대상 |
|------|------|----------|
| **구독** | 완전 관리형 Digital FTE 월 정액 ($500~2,000/월) | "핸즈오프" 자동화 고객 |
| **성과 수수료** | 결과 기반 커미션 (리드당 $5, 절감액의 2%) | 고신뢰, 이해관계 정렬 |
| **라이선스** | 독점 에이전트 로직 연 정액 | 데이터 주권 필요 기업 (의료, 금융, 국방) |
| **마켓플레이스** | 앱스토어를 통한 대량 판매 | 볼륨 플레이, 브랜드 구축 |

### Forward-deployed Engineer: GTM 전략

> "데모하지 말고 배포하라."

- **Forward-deployed engineer** = 고객 조직에 임베드되어 함께 솔루션을 구축하는 엔지니어
- Palantir가 발명 ("Deltas"와 "Echoes"). a16z는 "테크에서 가장 핫한 역할 중 하나"로 지목
- OpenAI와 Anthropic 모두 2026년 5월에 이를 공식화

**복리 루프:**
- 세일즈 주도 = 성사된 거래 + CRM 노트
- Forward-deployed = 성사된 거래 + **제품 일부** + 다음 배포를 빠르게 하는 도메인 지식

## 결론

천재나 큰 팀이 필요한 것이 아니다.

필요한 것은 **규율**:
1. 업무를 매핑하고
2. 컨텍스트를 쌓고
3. eval을 쓰고
4. 매주 루프를 돌린다

> 모두가 같은 모델을 쓰는 지금, **운영 체계가 비밀 무기**다.

회사를 코파일럿이 달린 계층이 아니라, **중심에 지능이 있고 엣지에 사람이 있는 시스템**으로 구축하라.
