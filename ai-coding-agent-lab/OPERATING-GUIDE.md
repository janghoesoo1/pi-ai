# AI Coding Agent Lab 운영 가이드

AI-네이티브 스타트업 프레임워크(cyberfund)를 이 실험실에 적용한 운영 가이드.

> 진짜 변화는 시간 사용이 아니라 학습하고 반복하고 진화하는 속도에 있다.
> 모두가 같은 모델을 쓰는 환경에서 진짜 해자는 규율이다.

---

## 프레임워크 요약

```
1. 업무 매핑       -> prompts/00-research.md + 자율성 레벨 분류
2. 컨텍스트 시스템  -> AGENTS.md + policies/ + context/
3. 자동화 선택     -> 스크립트 vs 워크플로 vs 에이전트 판단
4. 스킬 + eval    -> prompts/ (스킬) + experiments/ (eval)
5. 팀 AI-네이티브  -> 온보딩은 산출물로 종료
6. 피드백 루프     -> 매주 리뷰 -> 이너/아우터 루프 실행
```

## 1. 업무 매핑

### 이 실험실에서의 반복 업무

| 업무 | 자율성 레벨 | 대응 프롬프트 |
|------|------------|--------------|
| 코드베이스 구조 분석 | L3 (AI 실행 + 사람 감독) | `prompts/03-agents-md-generator.md` |
| N+1 쿼리 탐지 | L3 | `prompts/07-spring-work.md` |
| PR 리뷰 | L2 (AI 준비 + 사람 승인) | `prompts/08-pr-review.md` |
| 보안 점검 | L2 | `prompts/09-pr-check.md` |
| 테스트 생성 | L3 | `prompts/07-spring-work.md` |
| 실험 기록 작성 | L3 | `prompts/12-experiment-template.md` |
| 도구 비교 벤치마크 | L2 | `prompts/06-tool-benchmark.md` |
| MCP 연동 설계 | L1 (사람 전용) | `prompts/10-mcp-design.md` |
| 아키텍처 결정 | L1 | `prompts/14-plan-only.md` |

### 자율성 레벨 정의

| 레벨 | 규칙 | 이 실험실에서의 의미 |
|------|------|---------------------|
| L1 | 사람 전용 | 아키텍처 결정, MCP 설계, 보안 정책 수립 |
| L2 | AI 준비 + 사람 승인 | PR 리뷰, 보안 점검, 투자자/이해관계자 문서 |
| L3 | AI 실행 + 사람 감독 | 코드 분석, 테스트 생성, 실험 기록, N+1 탐지 |
| L4 | 한계 안에서 자율 | 린트, 포맷팅, 빌드 검증, 경쟁 도구 모니터링 |

### 첫 공략 대상 선택 기준

- 빈번하고
- 측정 가능하고
- 되돌릴 수 있고
- 실제 병목과 연결된 작업

> 지루한 워크플로가 보통 승리한다.
> 주간 전략 메모보다 매일 반복되는 코드 리뷰가 더 많은 시간을 회수한다.

## 2. 컨텍스트 시스템

### 이 실험실의 컨텍스트 구조

```
ai-coding-agent-lab/
  AGENTS.md              <- 에이전트 운영 규칙 (항상 로드)
  OPERATING-GUIDE.md     <- 이 문서 (운영 프레임워크)
  context/               <- 컨텍스트 레이어 (신규)
    lab-overview.md      <- 실험실 목적, 현재 상태, 핵심 결정
    tool-insights.md     <- 도구별 학습 사항 (실험 후 누적)
    decisions.md         <- 결정 로그 (append-only)
    lessons.md           <- 실패와 교훈 (append-only)
  prompts/               <- 스킬 (반복 작업 지침)
  policies/              <- 권한 경계
  experiments/           <- eval 기록
```

### 컨텍스트 3종 분류

| 종류 | 이 실험실에서의 대응 | 관리 규칙 |
|------|---------------------|-----------|
| 항상 로드 | `AGENTS.md`, `context/lab-overview.md` | 40~60줄 이내 유지. 잡문보다 촘촘한 목록 |
| 필요 시 로드 | `prompts/`, `policies/`, `context/tool-insights.md` | 작업별 선택 로드 |
| 절대 로드 금지 | `.env`, `.ssh/`, credentials | `policies/denied-commands.md`에 명시 |

### 주간 컨텍스트 진단법

대표 작업 하나를 워크스페이스 컨텍스트만 가진 새 에이전트에게 주고 다음 행동 3가지를 요청:
- 강한 제안 2개 이상 -> 컨텍스트 충분
- 일반적 답변 3개 -> 컨텍스트 빈약 (프롬프트로 구제 불가)

### 원본 vs 정제본 분리

| 구분 | 예시 |
|------|------|
| 원본 | 실험 전체 로그, git diff 전문, 도구 출력 원문 |
| 정제본 | experiments/ 기록 (점수, 핵심 발견, 교훈) |

에이전트가 조회하는 대상은 정제본이다. 원본은 검증이 필요할 때만 참조한다.

## 3. 자동화 선택

### 이 실험실에서의 도구 매핑

| 도구 유형 | 적합한 경우 | 실험실 예시 |
|----------|------------|------------|
| 스크립트 | 결정론적 단계 | `./gradlew test`, lint, build 검증 |
| AI 보조 사람 | 판단 필요한 출력 | PR 리뷰 결과 검토, 아키텍처 결정 |
| 워크플로 | 단계가 사전에 알려진 경우 | 코드 분석 -> 문제 탐지 -> 수정 제안 -> 테스트 -> 리뷰 |
| 에이전트 | 경로를 미리 정할 수 없는 경우 | 복잡한 버그 조사, 도구 간 비교 실험 |

### 하네스 6단계 적용

| 단계 | 이 실험실에서의 적용 |
|------|---------------------|
| Preflight | `AGENTS.md` + `policies/` 로드. 금지 명령 확인 |
| Plan | `prompts/14-plan-only.md`로 변경 계획 수립 |
| Approve | 사람이 계획 검토 후 승인 |
| Execute | `prompts/15-minimal-change.md`로 최소 변경 실행 |
| Verify | `prompts/08-pr-review.md` + `09-pr-check.md`로 검증 |
| Log | `experiments/YYYY-MM-DD-tool-task.md`에 기록 |

## 4. 스킬과 eval

### 스킬 = prompts/ 파일

이 실험실의 프롬프트 파일은 AI-네이티브 프레임워크의 "스킬"에 해당한다.

스킬에 필요한 형태:

| 항목 | prompts/ 파일에서의 대응 |
|------|-------------------------|
| Scope | frontmatter의 `usage` |
| Inputs | 프롬프트 내 "대상" 또는 "상황" 섹션 |
| Load | 프롬프트 내 "분석할 항목" 섹션 |
| Steps | 프롬프트 내 "진행 방식" 또는 "절차" |
| Output | 프롬프트 내 "출력 형식" 또는 "결과 형식" |
| Examples | experiments/ 기록 (실행 후 축적) |
| Escalate | 프롬프트 내 "제약" 또는 "중요 규칙" |
| Owner | 실험실 운영자 (장회수) |
| Eval | experiments/ 기록의 점수 기준 |

### eval = experiments/ 기록

#### eval 3개 레이어

| 레이어 | 이 실험실에서의 적용 |
|--------|---------------------|
| 손 라벨링 정답 | 실험자가 직접 점수 부여 (정확성, 변경 통제, 테스트 품질, 설명 품질, 안전성 각 1~5) |
| 결정론적 검사 | 빌드 성공 여부, 테스트 통과 여부, 변경 파일 수, 금지 명령 포함 여부 |
| LLM 판정 | AI 도구로 다른 AI 도구의 출력 품질 평가 (교차 검증) |

#### 매주 추적할 6개 숫자

| 지표 | 이 실험실에서의 의미 |
|------|---------------------|
| 수용률 | 에이전트 제안을 그대로 적용한 비율 (~70% 미만이면 스킬 개선 필요) |
| 오버라이드율 | 사람이 결과를 덮어쓴 비율 |
| 실행당 비용 | 토큰 사용량 / API 비용 |
| 사이클 타임 | 프롬프트 입력부터 결과 확정까지 시간 |
| 검토 시간 | 사람이 검토에 쓴 시간 |
| 사건 수 | 에이전트가 금지 명령 실행, 잘못된 파일 수정 등 사고 건수 |

> 수용률이 낮을 때 프롬프트 재작성은 거의 정답이 아니다.
> 보통: 컨텍스트 추가, 스킬 범위 좁히기, 작동 예시 추가, 에스컬레이션 규칙 작성 중 하나.

## 5. 팀 AI-네이티브

### 온보딩 = 산출물로 종료

새로운 팀원이 이 실험실에 합류할 때:

1. `README.md`와 `AGENTS.md`를 읽는다
2. `prompts/` 중 하나를 선택해 `sandbox/sample-spring-api`에 실행한다
3. 결과를 `experiments/YYYY-MM-DD-tool-task.md` 형태로 기록한다
4. 점수를 매기고 교훈을 `context/lessons.md`에 추가한다

> 실제 작업을 만들지 못하는 교육은 다음 주면 잊힌다.

### 역할 분리

| 사람의 역할 | 에이전트의 역할 |
|------------|---------------|
| 실험 설계 | 코드 분석, 문제 탐지 |
| 품질 판단 (eval 작성) | 반복 실행 |
| 아키텍처 결정 | 테스트 생성, 리포트 작성 |
| 도구 선택, 정책 수립 | 벤치마크 데이터 수집 |
| 결과 검증, 교훈 정리 | diff 생성, PR 설명 작성 |

## 6. 피드백 루프

### 주간 리뷰 체크리스트

매주 금요일, 다음 항목을 점검한다:

- [ ] 이번 주 실행한 실험과 결과 점수
- [ ] 에이전트 제안을 오버라이드한 지점과 이유
- [ ] 실패한 eval과 원인
- [ ] 누락된 컨텍스트 (context/ 업데이트 필요 여부)
- [ ] 좁혀야 할 스킬 (prompts/ 수정 필요 여부)
- [ ] 폐기할 워크플로
- [ ] 자율성 레벨을 올릴 워크플로
- [ ] 다음 주 실험 계획

### 이너 루프 (기존 작업 개선)

| 개선 대상 | 방법 |
|----------|------|
| 실행당 비용 | 컨텍스트 최적화, 불필요한 파일 로드 제거 |
| 사이클 타임 | 프롬프트 구체화, 단계 축소 |
| 사건 수 | policies/ 강화, 에스컬레이션 규칙 추가 |
| 검토 시간 | eval 자동화 (결정론적 검사 확대) |

### 아우터 루프 (새로운 탐색)

| 탐색 대상 | 방법 |
|----------|------|
| 새로운 AI 도구 | `prompts/01-tool-evaluation.md`로 평가 |
| 새로운 자동화 영역 | 업무 매핑 재실행 |
| MCP 연동 | `prompts/10-mcp-design.md`로 설계 |
| 제품화 | `prompts/13-product-design.md`로 기획 |

### 소프트웨어 팩토리 하드 룰

1. 어떤 것도 **자동 머지되지 않는다**
2. 어떤 에이전트도 **프로덕션에 쓰지 않는다**

> 이 게이트가 나머지를 안전하게 확장하게 한다.

## 핵심 원칙

1. **모델은 교체 가능하지만 컨텍스트와 eval은 회사만의 자산이다**
2. **eval 작성이 멈추는 순간 복리 성장도 멈춘다**
3. **가드레일은 코드와 설정에. 프롬프트 지시는 보안 경계가 아니다**
4. **출력 재작성에 검토보다 시간이 든다면 문제는 컨텍스트다**
5. **규율이 해자다 -- 매주 매핑하고, 쌓고, 평가하고, 돌린다**
